# コンソールRPG 実装課題 ― フェーズ4：Battleクラスで「進行」を切り出す

## このお題のゴール

フェーズ3までで、キャラクター側（`Character` / `Hero` / `Monster`）の設計はきれいになりました。
でも `main` を見てください。**キャラを作る・遭遇メッセージを出す・whileループを回す・勝敗を判定する**、これ全部が `main` の中に詰め込まれています。

このフェーズ4では、そこを分けます。`Battle` というクラスを新しく作り、**「戦いの進行」をまるごとそこへ引っ越し** させます。

このフェーズ4で身につく内容：

- **責務分離**（1つのクラスには1つの役割だけ持たせる）
- **クラスは「もの」だけでなく「仕事」も表せる**（`Battle` は勇者でもモンスターでもなく、"バトルの進行役"）
- **`private` メソッド**（クラスの中だけで使う、外に見せない処理）
- **パッケージを分ける**（`rpg.model` と `rpg.battle`）

> 新しい文法はほとんど出てきません。**すでに書いたコードを「どこに置くか」だけを直す** のが今回の本番です。動きを変えずに構造だけ良くすることを **リファクタリング** と言います。

---

## なぜ分けるのか

いまの `main` は「ゲームの組み立て」と「戦いのルール」が混ざっています。このままだと、こういうときに困ります。

- バトルのルールを変えたい（先攻後攻を決める、逃げるを足す）→ `main` を触ることになる
- あとで「タイトル画面 → バトル → 結果画面」と増やしたい → `main` がどんどん膨らむ
- バトルだけ動かして試したい → `main` を実行するしかない

そこで役割をこう分けます。

| クラス | 役割 | ひとことで言うと |
|---|---|---|
| `Main` | 登場人物を用意して、バトルを始める | **組み立てて、開始ボタンを押す人** |
| `Battle` | 戦いを最後まで進行させる | **試合の審判・進行役** |
| `Hero` / `Monster` | 攻撃する・倒れる | **選手** |

`main` が短くなって「何をしているか」が一目で読めるようになれば成功です。

## 作るもの（完成イメージ）

**画面の出力はフェーズ3と同じ**です。見た目は変えず、中身（構造）だけを良くします。

```
勇者 は スライム に遭遇した！
勇者 のHP: 30
スライム のHP: 15
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: 5
スライム の かみつき！ 勇者 に 4 のダメージ！
勇者 のHP: 26
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: -5
スライム は たおれた！

```

今回の成果は画面ではなく、**「`main` が数行になる」「バトルのルールが `Battle` 1か所に集まる」** という構造面に出ます。

## クラス構成

```
rpg/
├── Main.java          … 組み立てて、開始するだけ
├── battle/
│   └── Battle.java    … ★今回の新クラス。戦いの進行役
└── model/
    ├── Character.java … abstract（フェーズ3のまま）
    ├── Hero.java      … （フェーズ3のまま）
    └── Monster.java   … （フェーズ3のまま）
```

- `model` パッケージ（登場するもの）は **今回いっさい変更しません**。
- 新しく `rpg.battle` パッケージを作り、その中に `Battle` を置きます。

> なぜフォルダを分けるのか：`model`（登場するもの）と `battle`（進行のしくみ）は種類の違う仲間だからです。本棚をジャンルで分けるのと同じ感覚です。

## クラス仕様：`Battle`（新しく作る）

### 宣言

```java
package rpg.battle;

import rpg.model.Hero;
import rpg.model.Monster;

public class Battle
```

- **注意**：`Battle` は `rpg.battle` パッケージにあるので、別パッケージの `Hero` / `Monster` を使うには `import` が必要です。

### フィールド

| フィールド名 | 型 | 修飾子 | 意味 |
|---|---|---|---|
| `hero` | `Hero` | `private` | この戦いに参加している勇者 |
| `monster` | `Monster` | `private` | この戦いに参加しているモンスター |

> `Battle` は「勇者とモンスターを **持っている**」クラスです。フィールドに持たせておくことで、どのメソッドからも `this.hero` で参照できます。
> `private` にするのは、外から勝手に入れ替えられないようにするため（フェーズ3で学んだカプセル化の続き）。

### コンストラクタ

```java
public Battle(Hero hero, Monster monster)
```

- 受け取った2体を、それぞれフィールドに代入する。
- ここでは **まだ戦いません**。「この2人で戦う、という準備だけ」をします。

### メソッド

#### `public void start()`
- **やること**：バトルを最初から最後まで進行させる。中身は、いまの `main` に書いてある処理をそのまま持ってくる形になります。
  1. 遭遇メッセージを表示する（例：`勇者 は スライム に遭遇した！`）
  2. 両者の状態を `showStatus()` で表示する
  3. `while` ループで、両者が生きている間バトルを繰り返す
     1. 勇者がモンスターを攻撃する
     2. モンスターが倒れていたらループを抜ける
     3. モンスターが勇者を攻撃する
  4. 結果を表示する（`showResult()` を呼ぶ）
- **ヒント**：`hero` / `monster` は引数で受け取るのではなく、**フィールドに入っているもの** を使います。`start()` に引数は要りません。

#### `private void showResult()`
- **やること**：生き残っているほうを勝者として表示する。倒れたほうの「たおれた！」も表示する。
- **なぜ `private` なのか**：結果表示は「バトルの途中で内部的に呼ぶ処理」であって、`main` から直接呼ぶものではないからです。
  **外から使ってほしいのは `start()` だけ** ＝ 公開するのは `start()` だけ、という考え方です。
- **ヒント**：`isAlive()` を使って、どちらが生き残ったかを判定します。

> クラスを作るときは「**外から呼ばれる入口はどれか**」を意識してください。入口だけ `public`、それ以外は `private`。これがカプセル化です。

## クラス仕様：`Main`（大幅に短くする）

`main` の仕事は、もう「組み立てて、開始する」だけです。

1. `Hero` を1体、`Monster` を1体作る。
2. その2体を渡して `Battle` を作る。
3. `battle.start()` を呼ぶ。

これだけです。`while` ループも勝敗判定も、`main` からは **消えます**（`Battle` に引っ越したので）。

> `main` が3〜4行になったら成功です。「何が起きるゲームなのか」が、読んだ瞬間にわかる状態を目指してください。

## 実装の進め方（おすすめ手順）

**一気に消さない**のがコツです。動くコードを壊さないように、少しずつ引っ越します。

1. `src/rpg/battle/` フォルダを作り、空の `Battle` クラス（フィールド＋コンストラクタだけ）を書く。この時点でコンパイルが通ることを確認する。
2. `Main` で `Battle` を `new` してみる。まだ `start()` は呼ばない。`import rpg.battle.Battle;` を忘れずに。
3. `Battle` に `start()` を作り、**まず遭遇メッセージと `showStatus()` だけ** を移す。`main` からその部分を消して `battle.start()` を呼び、出力が変わらないことを確認する。
4. 次に `while` ループを `start()` に移す。実行して確認する。
5. 最後に勝敗判定を `showResult()` に移し、`start()` の最後から呼ぶ。実行して確認する。
6. `main` に余計なコードが残っていないか見直す。

各ステップで「コンパイル → 実行 → 出力が前と同じか確認」を挟むこと。
**リファクタリングは「出力が変わらないこと」が成功の証拠** です。

```bash
javac -d out src/rpg/**/*.java src/rpg/*.java
java -cp out rpg.Main
```
