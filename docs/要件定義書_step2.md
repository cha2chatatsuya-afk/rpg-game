# コンソールRPG 実装課題 ― フェーズ2：継承でキャラを分ける

## このお題のゴール

フェーズ1では、勇者もモンスターも同じ `Character` クラスから作っていました。
このフェーズ2では、`Character` を **親クラス** にして、`Hero`（勇者）と `Monster`（モンスター）という **子クラス** に分けます。そして、攻撃の仕方（`attack()`）をそれぞれ違う動きに変えます。

このフェーズ2で身につく内容：

- **継承**（`extends` で親クラスの機能を引き継ぐ）
- **`super(...)`**（子クラスのコンストラクタから親のコンストラクタを呼ぶ）
- **オーバーライド**（`@Override`。親のメソッドを子で作り直す）
- **`protected`**（子クラスから親のフィールドを使えるようにする）
- おまけで **ポリモーフィズム**（`.attack()` という同じ呼び出しが、相手の実体に応じて別々のメソッドに振り分けられる）

> **用語の整理**：*オーバーライド* は「子が `attack()` を自分用に **書き換える**」という **定義の話**。*ポリモーフィズム* は「同じ `.attack()` 呼び出しが、相手の実体に応じて **自動で振り分かる**」という **実行の話**。この2つは別の段階を指しています。

> フェーズ1で作った `Character` がそのまま土台になります。フィールドやコンストラクタは基本そのまま使い、**「共通部分は親に残す・違う部分は子に移す」** という仕分けが今回の本番です。
>
> `abstract`（抽象クラス）は **次のフェーズ3** で扱います。今回は `Character` を「そのまま new もできる普通の親クラス」として作ります。

---

## 作るもの（完成イメージ）

実行すると、勇者とモンスターで攻撃メッセージや威力が変わります（文言・数値は自由にアレンジしてOK）。

```
勇者 のHP: 30
スライム のHP: 15
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: 5
スライム の かみつき！ 勇者 に 4 のダメージ！
勇者 のHP: 26
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: -5
スライム は たおれた！
----------------------------------------
勇者 の勝ち！
```

ポイントは、**同じ「攻撃する」でも、勇者とモンスターで出るメッセージ・威力が違う**こと。これがオーバーライドの成果です。

---

## クラス構成

```
        Character (親クラス)
          ▲             ▲
          │ extends     │ extends
        Hero          Monster
```

- `Character` … 共通の土台（名前・HP・攻撃力と、共通の動作）。
- `Hero` / `Monster` … `Character` を継承し、`attack()` を自分用に作り替える。

---

## クラス仕様：`Character`（親クラス）

フェーズ1の `Character` を改造します。**共通で使うもの**を親に残し、攻撃の中身は子で上書きできるようにします。

### 宣言
```java
class Character
```
- フェーズ1と同じ普通のクラス。今回はまだ `abstract` は付けない。

### フィールド

| フィールド名 | 型 | 修飾子 | 意味 |
|---|---|---|---|
| `name` | `String` | `protected` | 名前 |
| `hp` | `int` | `protected` | 体力 |
| `attackPower` | `int` | `protected` | 攻撃力 |

> `private` から `protected` に変える理由：子クラス（Hero / Monster）から `this.attackPower` などを読めるようにするため。`private` だと子から見えません。

### コンストラクタ
```java
Character(String name, int hp, int attackPower)
```
- フェーズ1と同じ。3つの値をフィールドに代入する。

### メソッド

#### `void attack(Character target)`
- **やること**：フェーズ1と同じ「基本の攻撃」。相手（`target`）のHPを自分の `attackPower` 分だけ減らし、攻撃メッセージを表示する。
- これは **子が上書きするための土台** です。Hero / Monster はこのメソッドを `@Override` して、自分用の攻撃に作り替えます。

#### `boolean isAlive()`
- フェーズ1と同じ。`hp > 0` を返す。**共通なので親に置いたまま**。

#### `void showStatus()`
- フェーズ1と同じ。名前とHPを表示する。**共通なので親に置いたまま**。

---

## クラス仕様：`Hero`（勇者）

### 宣言
```java
class Hero extends Character
```

### コンストラクタ
```java
Hero(String name, int hp, int attackPower)
```
- 中で `super(name, hp, attackPower);` を呼ぶだけ。
- **ヒント**：`super(...)` はコンストラクタの一番最初の行に書く。

### メソッド

#### `@Override void attack(Character target)`
- **やること**：会心の一撃。ダメージは `attackPower + 2`（例）。
  1. ダメージ量を計算する（例：`attackPower + 2`）
  2. 「勇者 の会心の一撃！ ○○ に □ のダメージ！」と表示する
  3. 相手のHPを、そのダメージ分だけ減らす
- **ヒント**：メソッドの上に `@Override` を付けると、親のメソッドを正しく上書きできているかコンパイラが確認してくれる。

---

## クラス仕様：`Monster`（モンスター）

### 宣言
```java
class Monster extends Character
```

### コンストラクタ
```java
Monster(String name, int hp, int attackPower)
```
- `super(...)` を呼ぶだけ。

### メソッド

#### `@Override void attack(Character target)`
- **やること**：ふつうの攻撃。ダメージは `attackPower`。
  1. 「○○ の かみつき！ △△ に □ のダメージ！」など、モンスターらしいメッセージを表示する
  2. 相手のHPを、そのダメージ分だけ減らす

> Hero と Monster で `attack()` の中身が違う ＝ これがオーバーライド。**同じ名前のメソッドが、クラスごとに別の動きをする**のを体感するのが目的。

---

## `main` の流れ

`Main` クラスの `main` メソッドは、フェーズ1とほぼ同じ。ただし **作るクラスが変わる**。

1. `Hero` を1体、`Monster` を1体作る（例：`new Hero("勇者", 30, 8)` / `new Monster("スライム", 15, 4)`）。
2. 開始時の状態を両者とも `showStatus()` で表示する。
3. `while` ループで、両者が生きている間バトルを繰り返す：
   1. 勇者がモンスターを攻撃する（`hero.attack(monster)`）。
   2. モンスターが倒れていたらループを抜ける。
   3. モンスターが勇者を攻撃する（`monster.attack(hero)`）。
4. ループを抜けたら、生き残っているほうを勝者として表示する。

> **気づいてほしいこと**：`hero.attack(monster)` と `monster.attack(hero)` は、書き方は同じ「`.attack(相手)`」なのに、実際には別々のメソッドが動く。これがポリモーフィズム。

---

## 実装の進め方（おすすめ手順）

一気に直さず、小さく動かしながら進める。

1. まず `Character` のフィールドを `private` → `protected` に変える。この時点ではまだ動きは変わらない。
2. `Hero` クラスを作り、コンストラクタで `super(...)` を呼ぶだけにする。
3. `Hero` に `attack()` をオーバーライドして実装する。`main` を `Hero` を作る形に直して、まず勇者だけ動かしてみる。
4. `Monster` クラスを同じ要領で作る。
5. `while` ループでバトル全体をつなげ、完成イメージの出力になるか確認する。

各ステップで「コンパイル → 実行」を挟むのがコツ。
