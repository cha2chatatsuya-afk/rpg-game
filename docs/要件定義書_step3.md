# コンソールRPG 実装課題 ― フェーズ3：抽象クラスとカプセル化で設計を締める

## このお題のゴール

フェーズ2では `Character` を親クラスにして、`Hero` / `Monster` に分けました。
ただ、いまの `Character` は **そのまま `new Character(...)` できてしまいます**。でも「Character という存在そのもの」は戦いません。戦うのは勇者かモンスターです。

このフェーズ3では、そこを設計としてきちんと締めます。

このフェーズ3で身につく内容：

- **抽象クラス**（`abstract class`。「共通の型」だけ決めて、単体では `new` できないようにする）
- **抽象メソッド**（中身を書かず、「やり方は子が決める」とだけ宣言する）
- **カプセル化**（フィールドを直接いじらず、`takeDamage(...)` という窓口を通す）

> フェーズ2で作った `Character` / `Hero` / `Monster` をそのまま改造します。新しいクラスは増えません。**「親を締めて、ダメージの与え方を窓口に統一する」** のが今回の本番です。

---

## 作るもの（完成イメージ）

**画面の出力はフェーズ2と同じ**です。見た目は変えず、中身（設計）だけを良くします。

```
勇者 のHP: 30
スライム のHP: 15
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: 5
スライム の かみつき！ 勇者 に 4 のダメージ！
勇者 のHP: 26
----------------------------------------
勇者 の会心の一撃！ スライム に 10 のダメージ！
スライム のHP: -5
スライム は たおれた！
----------------------------------------
勇者 の勝ち！
```

今回の成果は画面ではなく、**「`new Character(...)` がコンパイルエラーになる」「ダメージ処理が1か所にまとまる」** という設計面に出ます。

## クラス構成

```
        Character (abstract / 抽象クラス)
          ▲             ▲
          │ extends     │ extends
        Hero          Monster
```

- `Character` … 共通の土台。**これ単体では new できない**（抽象クラスにする）。
- `Hero` / `Monster` … フェーズ2から変更なし（ただし `attack()` の中身を窓口経由に直す）。

---

## クラス仕様：`Character`（抽象クラスにする）

フェーズ2の `Character` を改造します。

### 宣言
```java
abstract class Character
```
- `abstract` を付けることで、`new Character(...)` を **禁止** する。
- 「Character という存在そのもの」は戦わない。戦うのは Hero か Monster。だから抽象にする。

### フィールド

フェーズ2から変更なし（`name` / `hp` / `attackPower` を `protected` のまま）。

### メソッド

#### `abstract void attack(Character target)`
- フェーズ2では「基本の攻撃」の中身が書いてありましたが、今回は **中身を消して抽象メソッドにする**。
- 「攻撃はできる。ただしやり方は子が決める」という宣言だけを残す。
- これがあるので `Character` は抽象クラスになる。

#### `void takeDamage(int damage)`  ← 今回の新メソッド
- **やること**：自分の `hp` を `damage` 分だけ減らす。
- 子クラスの `attack()` は、相手に対して `target.takeDamage(...)` を呼んでダメージを与える形にする（相手の `hp` を直接いじらず、この窓口を通す）。
- **なぜ窓口を作るのか**：「HPを減らす」という操作を1か所（`takeDamage`）にまとめると、あとで「防御力を引く」「0未満にしない」などのルールを足したくなったとき、ここだけ直せば済むから。これが **カプセル化** の考え方。

#### `boolean isAlive()` / `void showStatus()`
- フェーズ2から変更なし。**共通なので親に置いたまま**。

---

## クラス仕様：`Hero` / `Monster`

宣言・コンストラクタはフェーズ2から変更なし（`extends` と `super(...)` はそのまま）。
変えるのは `attack()` の **ダメージの与え方だけ**。

### `Hero` の `@Override void attack(Character target)`
- 会心の一撃。ダメージは `attackPower + 2`（例）。
  1. ダメージ量を計算する
  2. 「勇者 の会心の一撃！ ○○ に □ のダメージ！」と表示する
  3. **`target.takeDamage(そのダメージ)`** で相手のHPを減らす（← フェーズ2では直接減らしていた部分をここに変える）

### `Monster` の `@Override void attack(Character target)`
- ふつうの攻撃。ダメージは `attackPower`。
  1. モンスターらしいメッセージを表示する
  2. **`target.takeDamage(attackPower)`** で相手のHPを減らす

---

## `main` の流れ

フェーズ2と **同じ**。作るのは `Hero` と `Monster`。

1. `Hero` を1体、`Monster` を1体作る。
   - **注意**：ここで `new Character(...)` は書けなくなる（抽象クラスにしたので）。必ず `Hero` か `Monster` を作る。
2. 開始時の状態を両者とも `showStatus()` で表示する。
3. `while` ループで、両者が生きている間バトルを繰り返す：
   1. 勇者がモンスターを攻撃する。
   2. モンスターが倒れていたらループを抜ける。
   3. モンスターが勇者を攻撃する。
4. ループを抜けたら、生き残っているほうを勝者として表示する。

---

## 実装の進め方（おすすめ手順）

1. `Character` に `takeDamage(int damage)` を追加する。まだ誰も使っていないのでコンパイルは通る。
2. `Hero` / `Monster` の `attack()` を、相手のHP直接操作から `target.takeDamage(...)` に書き換える。実行して出力が変わらないことを確認する。
3. `Character` の `attack()` の中身を消して `abstract void attack(Character target);` にし、クラス宣言に `abstract` を付ける。
4. もし `main` などに `new Character(...)` が残っていればコンパイルエラーになる。`Hero` / `Monster` に直す。
5. 実行して、フェーズ2と同じ出力になるか確認する。

各ステップで「コンパイル → 実行」を挟むのがコツ。
